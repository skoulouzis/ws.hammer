Instructions for submission of advanced queries
Description of the options for the semantic matchmaker

The WS-HAMMER has an engine performing the semantic matchmaking task. This engine is based on OWLS-MX, which is a hybrid matchmaker. The process of matchmaking involves an OWL-S Profile as query. This query is basically a functional representation of the resource, in terms of IOPEs. The hybrid matchmaker combines the logic-based semantic matching with syntactic token-based similarity metrics, in order to obtain both, description logics and information retrieval of related resources.

Although WS-HAMMER supports six different techniques to discover resources, here only the five instances (M0 to M4) of the advanced discovery are explained. The keyword discovery of resources can be easily perceived.

Matchmaker algorithm
The five generic hybrid matching algorithms establish whether and which syntactic similarity metric is used by the WS-HAMMER's engine. This engine uses an OWL-DL description logic reasoner (Pellet) for logic based filtering, and the cosine, loss-of-information, extended Jacquard, and Jensen-Shannon information divergence based similarity metrics for complementary approximate matching.
In essence, each of algorithm uses the same logic-based semantic filters but different information retrieval similarity metric for content-based matching of the functional representation of the resource.
M0 : Purely logic-based semantic matching
The algorithm MO performs logic based only semantic matching of resources I/O description.
The logic-based semantic filters Exact, Plug-in, and Subsumes are applied as defined in the following Minimum degree section, whereas the hybrid filter Subsumed-by is utilized without checking the syntactic similarity constraint.
M1 : M0 + Loss-of-information similarity measure
M2 : M0 + Extended Jacquard similarity measure
M3 : M0 + Cosine similarity measure
M4 : M0 + Jensen-Shannon divergence based similarity measure
Minimum degree
The semantic matchmaking process computes the degree of semantic matching for a given pair of resource advertisement and request by successively applying five different filters: Exact, Plug-in, Subsumes, Subsumed-by and Nearest-neighbour.
The first three are logic-based only whereas the last two are hybrid due to the required additional computation of syntactic similarity values. Thus, if the M0 variant is selected, selecting nearest-neighbour as minimum is equal to selecting subsumed-by as there is no nearest neighbour filter for M0.
The order of matching filters regarding size of results due to relaxation of constraints is the following: Exact < Plug-In < Subsumes < Subsumed-By < Nearest-neighbour.
Exact : The funtional representation of the resource (I/O signature) perfectly matches the one from the request (with respect to logic-based equivalence of their formal semantics).
Plug-in : By relaxing the exact matching constraint, a resource may require less input than it has been specified in the request.
This guarantees at a minimum that the resource can be invoked with the provided input, if the involved OWL input concepts can be equivalently mapped to corresponding resource signature.
In addition, the resource is expected to return more specific output data whose logically defined semantics is exactly the same or very close to what has been requested by the user.
This kind of match is borrowed from the software engineering domain, where software components are considered to plug-in match with each other as defined above but not restricting the output concepts to be direct children of those of the query. Please note that Plug-in is a necessary constraint of the subsequent filters.
Subsumes : This filter is weaker than the Plug-in filter with respect to the extent the returned output is more specific than requested by the user. This happens since the filter relaxes the constraint of immediate output concept subsumption.
As a consequence, the returned set of relevant resources is extended in principle.
Subsumes-by : This filter selects resources whose output data is more general than one requested. In this sense, the filter subsumes the request. The matchmaker focuses on direct parent output concepts to avoid selecting resources returning data which might be too general.
It depends on the individual perspective taken by the user, the application domain, and the granularity of the underlying ontology at hand, whether a relaxation of this constraint is appropriate, or not.
Nearest-neigbour : TBM
Syntactic similarity threshold
This slider allows the users to set the value of the similarity threshold to appropriate levels. The threshold value is used by the hybrid filters (subsumed-by and nearest neighbour).
On one hand, if the semantic degree of a service is subsumed-by but the syntactic similarity is below the threshold, then the hybrid degree of match is fail. On the other hand, if the semantic degree is fail but the syntactic similarity is higher than, or equal to the threshold, then the resulting hybrid degree of match is nearest neighbour.
If a resource does not match with request, according to any of the filters, the semantic degree is fail.
